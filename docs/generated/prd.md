# PRD: Qwen3-TTS macOS 原生客户端（本地推理）

> 生成于 2026-07-10，由 /forgeflow:to-prd 产出。MVP 导向，作为拆解 issue 和实现的工程输入。
> 已确认决策：**只做本地推理，不接云端 API**（用户明确要求）。
> 事实依据：Qwen3-TTS 已于 2026-01-22 开源（Apache 2.0，0.6B/1.7B，Base / CustomVoice / VoiceDesign 变体），Apple Silicon 上有 MLX 和 CoreML 两条已验证的本地推理路径。

## 问题陈述

想在 Mac 上使用 Qwen3-TTS 的用户，目前只能通过命令行工具（mlx-audio、Python 脚本）本地运行：需要配 Python 环境、手动下载权重、无 GUI、无法方便地试听和管理结果。缺少一个开箱即用、完全离线、不需要 API Key 和联网付费的原生客户端。

## MVP 目标

用户首次启动 App，在引导下完成模型下载后，完成一次「输入文本 → 选择预置音色 → 点击合成 → 边合成边听 → 导出音频文件」的完整闭环，全程无需网络（模型下载后）、无需 API Key。合成首包延迟在 Apple Silicon 上足够低（目标 < 2 秒）即认为 MVP 成立。

## 目标用户

- 主要用户：持有 Apple Silicon Mac、重视隐私或离线场景、需要把文本转成高质量语音的个人用户（内容创作者、开发者自用）。
- 次要用户：想在本机快速试听 Qwen3-TTS 各音色效果、不想折腾 Python 环境的开发者。

## 解决方案

一个 SwiftUI 原生 macOS 应用，单窗口、单一职责：只服务 Qwen3-TTS 本地推理。首次启动引导用户选择并下载模型（含磁盘与内存占用提示），之后主界面就是「文本框 + 音色选择 + 合成按钮」。合成在本机 Apple Silicon 上流式执行，音频生成即开始出声；完成后可重播、导出为文件。历史记录保留最近合成结果，可重放或重新导出。

## 用户故事

1. 作为一个内容创作者，我想要输入一段文案并立即在本机听到合成语音，以便快速迭代文案的听感，且内容不离开我的电脑。
2. 作为一个用户，我想要浏览并试听全部预置音色，以便为我的内容挑选合适的声音。
3. 作为一个用户，我想要把满意的合成结果导出为音频文件，以便在剪辑软件中使用。
4. 作为一个用户，我想要 App 引导我下载和管理模型文件，以便不用手动折腾 Hugging Face 和命令行。
5. 作为一个用户，我想要 App 记住我上次使用的音色和参数，以便下次打开即用。

## 范围

### 范围内

- 首次启动引导：模型选择（默认推荐一档）、下载、断点续传、完整性校验、磁盘/内存占用提示。
- 模型管理：查看已下载模型、删除、切换。
- 文本输入（支持长文本粘贴，展示字符数），预置音色选择（官方 9 个 speaker），记住上次选择。
- 本地流式合成与边到边播（低首包延迟），播放控制（播放/暂停/停止/重播），合成中途可取消。
- 导出音频文件（wav，24 kHz）。
- 本地历史记录：最近合成的文本 + 参数 + 音频缓存，可重放和导出。
- 错误处理：磁盘不足、内存不足、下载失败、不支持的机型（Intel Mac），均有可理解的提示。

### 范围外

- 任何云端 API 调用（含 DashScope）——本项目永久范围外。
- 音色克隆（CustomVoice）与语音设计（VoiceDesign）——模型能力支持，留到下一轮。
- 除 Qwen3-TTS 以外的任何模型。
- Intel Mac 支持、iOS / iPadOS、App Store 上架流程。
- 批量任务队列、账号系统、云端同步。

## 实现决策

- 主要 Module：
  - **推理引擎（InferenceEngine）**：全 App 最深的模块。隐藏推理运行时的全部细节（MLX 或 CoreML、tokenization、KV cache、音频 codec 解码、流式分帧）；对外只暴露「给文本和音色，返回音频块流」的语义。运行时选型是最大的变化点，必须被这个接口完全隔离。
  - **模型管理（ModelManager）**：隐藏 Hugging Face 下载协议、断点续传、文件校验与磁盘布局；对外是「列出可用/已下载模型、下载（带进度）、删除、就绪状态」。
  - **播放引擎（AudioPlayer)**：隐藏流式缓冲与 AVFoundation 细节；对外是「喂入音频块、播放控制、进度回调」。
  - **历史记录（History）**：隐藏音频缓存文件管理与元数据持久化；对外是增查删。
- 关键 Interface：推理引擎以异步流（AsyncSequence 语义）交付 PCM 音频块，播放引擎消费同一流。取消语义（用户中途停止）必须正确释放推理任务和内存。
- 数据或 API 契约：模型权重来自 Hugging Face（Qwen 官方或 mlx-community 量化版）；音频输出 24 kHz PCM/wav；音色枚举以官方 9 个 speaker 为准，做成数据驱动。
- 架构约束：SwiftUI + Swift Concurrency，仅支持 Apple Silicon，macOS 14+；推理运行时选型（见开放问题）确定前，推理引擎接口先行冻结。
- 已有 ADR 对齐：无。**推理运行时选型是高影响 trade-off，需用 decision-forge 收敛并沉淀为首个 ADR。**

## 测试决策

- 关键路径：文本 → 推理引擎产出音频块流 → 播放引擎出声 → 导出文件字节正确。推理引擎接口用 fake 实现（预录音频块）驱动上层测试，不依赖真实模型。
- 边界行为：空文本（禁用合成）、长文本分段策略、合成中途取消（流终止、推理任务释放、无内存泄漏）、模型未下载时的引导态。
- 异常行为：下载中断恢复、磁盘写满、模型文件损坏（校验失败重下）、内存压力下的降级或明确报错。
- 可复用测试先例：无（新项目）；推理引擎的 fake 实现将成为全项目测试基座。

## 风险与开放问题

- ~~推理运行时选型~~ **已决策（ADR-0001）**：MLX Swift（mlx-audio-swift）。实测 TTFB 0.11~0.13s、RTFx 3.4~4.1（M3 Max，8bit），远超 MVP 标准；默认模型 0.6B-CustomVoice-8bit，1.7B 作音质选项。数据见 docs/generated/spike-inference-runtime.md。
- **依赖成熟度**：mlx-audio-swift 为 v0.1.x 早期库，靠 InferenceEngine 接口隔离兜底；升级或替换实现的变化被限制在 TTSEngineMLX 模块内。
- **模型下载体验**：0.6B 实测约 1.8 GB、1.7B 约 2.9 GB（远大于早期估算），首次启动下载时长与失败恢复是第一印象的关键路径；需断点续传与完整性校验（spike 中网络中断反复出现，是真实场景）。
- **长文本策略**：spike 中约 220 字一次生成 51s 音频无问题；更长文本的分段阈值与段间韵律衔接待实测确定。
- **低配机型表现**：全部数据来自 M3 Max/64GB；8GB 内存机型上 0.6B（约 2GB 常驻）的表现待验证。
